13 Ograniczenia Testow — Feedback CTO

Po publikacji v2 Alex (CTO / co-founder) zglosil krytyczne uwagi dot. designu testow. Ponizsze ograniczenia MUSZA byc uwzglednione przy interpretacji wynikow.

1. Brak JSON Schema w API call = sztuczne JSON failures ROZWIAZANE

Testy uzywaly text: {"format": {"type": "text"}} zamiast JSON schema output format (response_format: {"type": "json_object"} lub Structured Outputs). W produkcji codebase WYMUSZA JSON schema — model MUSI zwrocic valid JSON.

Rozwiazanie: JSON schema validity (0-2 pkt) usunieta z rubric. Skala zmieniona z 15 → 13 punktow. 2 catastrophic failures GPT-5.4 LOW (OBJECTION_TIME, SILENT_USER) wykluczone z oceny jako artefakt testu. Srednia 5.4 LOW przeliczona na 23 konwersacji (bez wykluczonych).

Wplyw: JSON nie jest juz kryterium. Wszystkie score'y przeliczone na skale /13. 5.4 LOW: 11.3/13 (86.6%) na 23 konwersacjach.

2. Brak Calendly prompt i tool calling

Testy ladowaly TYLKO main_system_current.md jako system prompt. W produkcji kazda organizacja ma dodatkowe zmienne: {{calendly_prompt}}, {{additional_info}}, oraz valid tool calling (np. booking trigger, form links).

Bez Calendly prompt model nie ma instrukcji dot. bookingu — wyniki BOOKING, CONVERTED, FORMSENT moga byc zanizone dla WSZYSTKICH konfiguracji (nie tylko 5.4).

3. Brak realnej obslugi obrazow i mediow

Obrazy/audio/stickery sa zastapione placeholderami ([User sent an image]). W produkcji codebase obsluguje te typy inaczej (np. image description via vision API). Wyniki IMAGE, AUDIO, NON_TEXT sa wazne dla text-handling, ale NIE odwzorowuja pelnego flow produkcyjnego.

4. Testy per-organizacja vs static conversations

Alex rownoczesnie uruchomil wlasne testy "per organizacja" na 5.4, z pelnym kontekstem (Calendly, additional_info, org-specific variables). Jego wyniki moga roznic sie od tego raportu ze wzgledu na bogatszy kontekst.

5. Booking codebase fixes nie uwzglednione

Alex fixowal booking flow w codebase (response handling, JSON parsing). Testy byly offline (nie przez produkcyjny backend) — nie korzystaly z tych fixow. Realne booking performance moze byc LEPSZE niz raportowane tutaj.
