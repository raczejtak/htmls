1 Executive Summary

Problem

Setor AI ma 4-16 tygodni przewagi konkurencyjnej zanim ktoś wejdzie z podobnym produktem za $200/mo. Obecny affiliate system (10→20→30%) nie jest wystarczająco agresywny żeby stworzyć efekt lawiny.

Rozwiązanie

Przerobić system na 40%-cap tiered model z obowiązkową drugą linią (T2 = 5%), wzorowany na ClickFunnels/GoHighLevel. Affiliate program staje się #1 growth engine.

Dlaczego TERAZ

Okno konkurencyjne 4-16 tygodni. Każdy dzień opóźnienia = mniej czasu na zbudowanie sieci affiliate zanim competitor zaoferuje tańszy produkt. Affiliate network raz zbudowany jest moat — trudny do skopiowania.

Kluczowe zasady architektury

  • Flat + 1 level — to NIE jest MLM. Max głębokość = 2 (affiliate → sub-affiliate → klient). Nigdy T3+.
  • T2 = stałe 5% — niezależnie od tiera. Starter, Growth, Elite, Legend — zawsze 5% z drugiej linii.
  • T2 odblokowane od dnia 1 — nawet Starter z 1 poleceniem ma T2, jeśli jego polecony też poleci kogoś.
  • Non-retroactive — stawka T1 locked w momencie rejestracji klienta. Tier up = nowe referrale mają wyższą stawkę, stare nie.
  • Cap 40% — T1 (max 35%) + T2 (5%) = max 40% od jednego klienta trafia do affiliate network.

Zmiana vs stary system: Starter = LIFETIME

W starym systemie Starter miał 12-miesięczny limit prowizji. W nowym systemie WSZYSTKIE tiery mają LIFETIME prowizje. Usunięcie 12-mies. expiry. Zmień isLifetimeCommission na true ZAWSZE.

Jak działa tier upgrade — NON-RETROACTIVE

Scenariusz: Affiliate ma 2 klientów (Starter, 10%). Poleca 3. klienta → wskakuje na Growth.

  • Klient #1 — locked 10% (Starter rate z momentu jego signup) → zostaje 10% na zawsze
  • Klient #2 — locked 10% → zostaje 10%
  • Klient #3 — locked 15% (Growth rate) → nowa stawka od tego momentu
  • Klient #4, #5 — locked 15%
  • Klient #6 — locked 25% (Elite rate)

Konsekwencja: Ten sam affiliate może mieć klientów na 10%, 15%, 25%, i 35% jednocześnie. Każdy na innej stawce z momentu rejestracji.

4 Druga Linia (T2) — Jak działa 5%

Definicja

T2 (Tier 2 / Druga Linia) = 5% prowizji od klientów poleconych przez TWOICH poleceń.

Jeśli Jasion polecił Klienta A, a Klient A polecił Klienta C → Jasion dostaje 5% od płatności Klienta C.

Warunki:

  • T2 odblokowane od dnia 1 — nawet Starter.
  • T2 = stałe 5% niezależnie od tiera.
  • Max głębokość = 1 poziom. Nigdy T3.
  • Cap: T1 + T2 ≤ 40% na jednego klienta.

Diagram T2 — krok po kroku

KROK 1: Jasion poleca Klienta A (sam się zapisuje do Setor) Jasion → parentAffiliate Klient A → staje się też affiliate (sub-affiliate Jasiona) Relacja: Klient A.parentAffiliateId = Jasion.affiliateId KROK 2: Klient A poleca Klienta C Klient A → affiliate (T1) Klient C → nowy klient Setor KROK 3: Klient C płaci $1,000/mies → System nalicza T1 dla Klienta A (np. 10% = $100) → System SPRAWDZA: czy Klient A ma parentAffiliateId? → TAK → parentAffiliateId = Jasion → System nalicza T2 dla Jasiona (5% = $50) KROK 4: Klient C poleca Klienta D → Klient C NIE MOŻE generować T2 dla Jasiona! → T2 = max 1 poziom głębokości → Jasion dostaje T2 TYLKO z bezpośrednich poleceń Klienta A → Jeśli Klient C jest affiliate → jego parent = Klient A (nie Jasion)

T2 — Kto co dostaje (tabela)

Klient Polecony przez T1 → komu T2 → komu
Klient A Jasion Jasion (T1) — (Jasion nie ma parenta)
Klient C Klient A Klient A (T1) Jasion (T2 = 5%)
Klient D Klient C Klient C (T1) Klient A (T2 = 5%) — NIE Jasion!
Klient E Klient D Klient D (T1) Klient C (T2 = 5%) — chain max 1 deep

HARD RULE: Max głębokość T2 = 1

Jasion → Klient A → Klient C → Klient D

Jasion dostaje T2 TYLKO z poleceń Klienta A. NIE z poleceń Klienta C. To NIE jest MLM piramida.

Implementacja: parentAffiliateId dotyczy BEZPOŚREDNIEGO parenta. Przy naliczaniu T2 sprawdź TYLKO affiliate.parentAffiliateId, nigdy nie rób lookup wyżej.

5 Kompletny Flow — Od kodu do kasy

Flow A: Rejestracja referrala

1. Affiliate dostaje status "approved" + kod (np. JASION7X) 2. Udostępnia link: https://app.setor.ai/ref/JASION7X 3. Nowy user klika link → /ref/[code]/page.tsx → redirect do /auth/sign-up?ref=JASION7X → NOWE: Set cookie "setor_ref=JASION7X" (90 dni, domain=.setor.ai) 4. User rejestruje się (signup) → validateReferralCode("JASION7X", orgId) → createReferralFromCode("JASION7X", orgId) → Tworzy AffiliateReferral: { affiliateId: jasion_aff_id, referredId: new_org_id, referralCode: "JASION7X", isActive: false, // jeszcze nie płaci commissionRate: 10, // locked na stawkę z momentu signup isLifetimeCommission: true // NOWE: zawsze true } 5. Referral CZEKA na pierwszą płatność (isActive: false)

Flow B: Pierwsza płatność → Aktywacja

1. Klient kończy trial → Stripe charge succeeded 2. Webhook invoice.paid → packages/billing/src/webhook.ts 3. activateReferral(orgId, chargeId) → isActive = true → firstPaidAt = now → firstPaymentChargeId = chargeId → Jeśli to 1. płacący referral tego affiliate → isFirstReferralForReferrer = true 4. Referral jest teraz AKTYWNY 5. 30-dniowy delay startuje (prowizja dopiero po 30 dniach)

Flow C: Naliczanie prowizji (miesięcznie)

1. Cron: /api/cron/affiliate-commissions (miesięcznie) 2. Dla każdego affiliate z aktywnymi referralami: a. getCommissionEligibleReferrals(affiliateId) → Filtruje: isActive=true, firstPaidAt > 30 dni temu b. Dla każdego eligible referrala: → hasMonthlyCommissionForMonth(affId, refId, "2026-03")? → skip jeśli tak → Pobierz aktualną cenę subskrypcji klienta → commissionCents = subscriptionPrice * commissionRate / 100 → recordAffiliateEarning({ type: "monthly_commission", amountCents: commissionCents, month: "2026-03" }) 3. NOWE — T2 Commission: → Po naliczeniu T1 dla affiliate X → Sprawdź: affiliate X ma parentAffiliateId? → TAK → nalicz T2 (5%) dla parenta: recordAffiliateEarning({ type: "tier2_commission", // NOWY typ amountCents: subscriptionPrice * 5 / 100, month: "2026-03", affiliateId: parentAffiliateId // parent dostaje T2 }) → NIE → skip T2

Flow D: Wypłata / Aplikacja na fakturę

1. Affiliate z organizationId (klient Setor): → Balance automatycznie odliczany od następnej faktury → calculateCreditsToApply() → Stripe coupon → debitCreditsFromBalance() 2. Affiliate BEZ organizationId (zewnętrzny): → Balance rośnie → Gdy ≥ $100 → może złożyć AffiliatePayoutRequest → Admin (CSV + przelew) na start → Stripe Connect później → payoutMethod: "bank_transfer" | "invoice_credit"

8 Churn, Grace Period, Dezaktywacja

KLIENT AKTYWNY → płaci → prowizja naliczana │ ▼ (anuluje/payment fails) DEACTIVATED → isActive=false, deactivatedAt=now │ ├── Wraca w ciągu 45 dni? → REAKTYWACJA (isActive=true, deactivatedAt=null) │ └── NIE wraca 45+ dni? → REFERRAL PERMANENTNIE MARTWY Nie liczy się do tiera Nie generuje prowizji Nie może być reaktywowany NIGDY

Impact na tier

Affiliate ma 6 aktywnych (Elite) → 2 churnują → spada do 4 aktywnych (Growth).

Istniejące stawki locked: klienci na 25% zostają na 25%. Ale NOWI klienci dostają Growth rate (15%).

Jeśli 2 wrócą w grace period → tier wraca do Elite. Nowi klienci od tego momentu = 25%.

10 Migracja — Stary vs Nowy system

Zasada: Nowe stawki TYLKO na NOWE referrale

  • Istniejące AffiliateReferral rekordy NIE są modyfikowane
  • Stare referrale zachowują swoje locked commissionRate i commissionAmountCents
  • Nowe referrale (od momentu deploy) dostają nowe stawki z getCommissionForPosition()
  • Jedyna zmiana na starych: isLifetimeCommission = true dla WSZYSTKICH (koniec 12-mies. limitu)

Migracja step-by-step

1. Deploy nowego getCommissionForPosition() → nowe stawki 10/15/25/35 2. Jednorazowy UPDATE na starych Starter referralach: UPDATE AffiliateReferral SET isLifetimeCommission = true WHERE isLifetimeCommission = false; 3. NIE modyfikuj commissionRate na starych referralach 4. Sync parentAffiliateId w schema.prisma (kolumna już w DB!) 5. Deploy T2 logic → działa od pierwszego crona po deploy

2.2 registerSubAffiliate()

export async function registerSubAffiliate(
  parentAffiliateId: string,
  childAffiliateId: string
): Promise<{ success: boolean; error?: string }> {
  // Guard 1: Parent exists and is approved
  const parent = await prisma.affiliate.findUnique({
    where: { id: parentAffiliateId },
    select: { status: true },
  });
  if (!parent || parent.status !== 'approved') {
    return { success: false, error: 'Parent affiliate not found or not approved' };
  }

  // Guard 2: No self-referral
  if (parentAffiliateId === childAffiliateId) {
    return { success: false, error: 'Cannot be your own sub-affiliate' };
  }

  // Guard 3: Child has no parent yet (atomic)
  const updated = await prisma.affiliate.updateMany({
    where: { id: childAffiliateId, parentAffiliateId: null },
    data: { parentAffiliateId },
  });

  if (updated.count === 0) {
    return { success: false, error: 'Already has a parent affiliate' };
  }

  return { success: true };
}

13 40 Edge Cases — Kompletna lista

A. Rejestracja i kody

1 Nieistniejący kod referralny
User wpisuje "ABCDEF99" który nie istnieje w bazie.
Rozwiązanie: validateReferralCode() → isValid: false, error: 'Invalid referral code'. Brak rekordu. User rejestruje się bez affiliate.
2 Self-referral (własny kod)
Affiliate próbuje użyć swojego kodu na swojej organizacji.
Rozwiązanie: Guard: affiliate.organizationId === currentOrgId → block. Komunikat: "Cannot use your own referral code".
3 Kod suspended/rejected affiliate
Affiliate został zawieszony ale ktoś ma jego link.
Rozwiązanie: affiliate.status !== 'approved' → block. Taki sam error jak nieistniejący kod (security through obscurity).
4 Org już ma przypisany kod + próbuje dodać drugi
Klient zarejestrował się z kodem ANNA1234, teraz chce zmienić na BARTEK56.
Rozwiązanie: canAddReferralCode() sprawdza org.referredByCode — jeśli istnieje → false. Jeden kod na org, na zawsze.
5 Kod po pierwszej płatności
Klient płaci, a dopiero potem wpisuje kod referralny.
Rozwiązanie: canAddReferralCode() sprawdza org.subscriptions.length === 0. Po płatności nie można dodać kodu.
6 Dwa signupy z tym samym kodem w tej samej sekundzie
Race condition — dwa nowe org'y używają kodu JASION7X jednocześnie.
Rozwiązanie: Każdy w osobnej prisma.$transaction. Pozycja (#1 vs #2) ustalana przez count + 1 w transakcji — izolacja gwarantowana.
7 Cookie 89. dzień vs 91. dzień
User klika link affiliate, wraca po 89 lub 91 dniach.
Rozwiązanie: Cookie maxAge: 90 days. Dzień 89 = cookie istnieje → atrybucja. Dzień 91 = cookie expired → brak atrybucji. Przeglądarka automatycznie kasuje.
8 Query param ?ref= vs cookie — konflikt
User ma cookie od affiliate A, ale klika link affiliate B z ?ref=B.
Rozwiązanie: Query param NADPISUJE cookie: const code = refParam || refCookie. Explicit > implicit. Cookie też nadpisany.
9 Cookie domain .setor.ai vs app.setor.ai
Cookie set na setor.ai, ale signup jest na app.setor.ai.
Rozwiązanie: Ustaw domain = .setor.ai (z kropką na początku) → widoczny na obu subdomenach.

B. Płatności i aktywacja

10 Polecony nigdy nie płaci (trial dropout)
Klient rejestruje się z kodem, robi trial, nie płaci.
Rozwiązanie: Referral zostaje isActive: false na zawsze. Zero prowizji. Nie liczy się do tiera. Nie generuje T2.
11 Refund przed 30 dniami (bonus $500)
Klient płaci, refunduje po 15 dniach.
Rozwiązanie: handleReferralRefund() → firstPaymentRefundedAt = now. Bonus NIGDY nie naliczony. Status: refunded_before_bonus.
12 Refund PO 30 dniach (clawback)
Bonus $500 już naliczony, klient robi refund.
Rozwiązanie: bonusClawbackRequired = true. Przy następnej fakturze affiliate: $500 odejmowane z balance. Status: clawed_back.
13 Race: 2 webhooks naliczają bonus jednocześnie
Dwa procesy jednocześnie widzą bonus jako eligible.
Rozwiązanie: Atomic guard: updateMany({ where: { bonusAppliedAt: null } }). Tylko updated.count > 0 = wygrana. Drugi dostaje count === 0 → skip.
14 Podwójna prowizja w tym samym miesiącu
Cron odpala się 2x w marcu (np. bug w scheduler).
Rozwiązanie: Unique constraint: @@unique([affiliateId, referralId, month]). Drugi INSERT → DB error → skip. Plus hasMonthlyCommissionForMonth() check.
15 Prowizja w miesiącu naliczenia bonusu
Bonus $500 naliczony w marcu, cron też chce naliczyć prowizję za marzec.
Rozwiązanie: Check: bonusAppliedAt month === current month → skip monthly commission. W miesiącu bonusu prowizja nie jest naliczana.

C. Churn i grace period

16 Klient anuluje → wraca po 20 dniach
Klient cancels, w grace period (45 dni), wraca i płaci.
Rozwiązanie: activateReferral() widzi deactivatedAt < 45 dni → reaktywuje: isActive=true, deactivatedAt=null. Prowizje wracają.
17 Klient anuluje → wraca po 50 dniach
Klient wraca po upływie 45-dniowego grace period.
Rozwiązanie: daysSinceDeactivation > 45 → referral permanentnie martwy. Nawet jeśli klient zapłaci. Żadna prowizja, żaden tier point.
18 Tier drop po churn
Affiliate miał 6 aktywnych (Elite), 2 churnują → 4 aktywnych (Growth).
Rozwiązanie: Tier spada na dashboardzie. Istniejące stawki NIE zmieniają się (non-retroactive). Nowi klienci dostają Growth rate (15%).
19 Klient w grace period — liczy się do tiera?
Klient deactivated ale w grace period 45 dni.
Rozwiązanie: NIE liczy się — isActive: false. Tier obliczany TYLKO z isActive: true. Grace period to szansa na powrót, nie gwarancja.

D. T2 (Druga Linia) — Edge Cases

20 Chain depth > 1 (T3 attempted)
Anna → Bartek → Celia → Dana. Anna chce T2 z Dana.
Rozwiązanie: T2 = TYLKO 1 poziom. Przy naliczaniu T2 sprawdź affiliate.parentAffiliateId — to BEZPOŚREDNI parent. Anna dostaje T2 z klientów Bartka, NIE z klientów Celii.
21 Circular: A → B → A (affiliate rekrutuje się nawzajem)
Affiliate A chce być sub-affiliate B, i B sub-affiliate A.
Rozwiązanie: registerSubAffiliate() → guard: updateMany WHERE parentAffiliateId IS NULL. Pierwszy call wygrywa. Drugi → count === 0 → "Already has a parent".
22 Self-sub-affiliate
Affiliate próbuje zarejestrować sam siebie jako sub-affiliate.
Rozwiązanie: Guard: parentAffiliateId === childAffiliateId → block. "Cannot be your own sub-affiliate".
23 T2 idempotency — cron przetwarza parent i child osobno
Cron przetwarza affiliate A (T1 credit) i zaraz potem przetwarza parent B (T2 credit). Ale T2 już został naliczony w kroku A.
Rozwiązanie: T2 naliczane RAZEM z T1 (po T1 credit → check parentAffiliateId → credit T2). Plus dedup: findFirst({ type: 'tier2_commission', affiliateId, referralId, month }).
24 Parent affiliate suspended → T2
Parent affiliate został zawieszony (np. fraud). Jego sub-affiliates dalej generują sprzedaż.
Rozwiązanie: Przed naliczeniem T2: parent.status === 'approved'. Jeśli suspended → skip T2. T1 dla sub-affiliate dalej naliczane normalnie.
25 Sub-affiliate churnuje jako klient ale dalej ma referrale
Klient A (sub-affiliate Jasiona) anuluje swoją subskrypcję Setor, ale jego poleceni dalej płacą.
Rozwiązanie: Affiliate status i klient status to OSOBNE rzeczy. Klient A dalej jest affiliate ze statusem approved. Jego referrale dalej generują T1 (dla A) i T2 (dla Jasiona). Churn subskrypcji ≠ deaktywacja affiliate.
26 Affiliate bez organizationId chce T2
Zewnętrzny affiliate (influencer) rekrutuje sub-affiliates.
Rozwiązanie: T2 działa niezależnie od organizationId. Zewnętrzny affiliate zbiera T2 na balance → payout via bank transfer.

E. Finansowe

27 40% cap exceeded
Bug/edge: T1 (35%) + T2 (5%) = 40% ok. Ale co jeśli ktoś manipuluje DB?
Rozwiązanie: Hard guard w recordAffiliateEarning(): oblicz totalCommissionRate = T1 + T2 per klient. Jeśli > 40% → reject/log/alert. Validation layer, nie tylko DB constraint.
28 Annual payment — duża jednorazowa prowizja
Klient płaci $10,000 (annual). Legend affiliate: 35% = $3,500 jednorazowo.
Rozwiązanie: Prowizja naliczana od FAKTYCZNEJ kwoty faktury. To jest by design — annual payment = annual commission. Cash flow impact do monitorowania. 30-day delay dalej obowiązuje.
29 Chargeback po wypłacie prowizji
Klient robi chargeback 3 miesiące po pierwszej płatności. Prowizje już wypłacone.
Rozwiązanie: chargeId w AffiliateEarning → pozwala na clawback konkretnej prowizji. Negative earning record. Jeśli balance za niski → "dług" (negative balance) odejmowany od przyszłych earnings.
30 Clawback $500 + balance = $0
Affiliate ma $300 balance, przychodzi clawback $500.
Rozwiązanie: creditBalanceCents spada do -$200. Negative balance → następne prowizje najpierw pokrywają dług. Affiliate nie dostaje nic dopóki balance > 0.

Klient upgrade z $1,000 na $2,000, potem downgrade na $1,000.
Rozwiązanie: Prowizja = % od FAKTYCZNEJ kwoty faktury (z Stripe subscription.items[0].priceAmount). Auto-adjust. Upgrade = wyższa prowizja. Downgrade = niższa.

Zewnętrzny affiliate ma $80 balance, chce wypłatę.
Rozwiązanie: AffiliatePayoutRequest → walidacja: if (balance < 10000) reject. Komunikat: "Minimum payout: $100. Current balance: $80."

36 Affiliate link spam / abuse
Bot generuje 10,000 requestów na /ref/[code].
Rozwiązanie: Rate-limit endpoint /ref/[code]: max 60 req/min per IP. Cookie set nie powoduje kosztów — redirect jest lekki. Signup wymaga email verification → fraud blocked na poziomie auth.
